Why TeamBench?

The First Structurally Enforced
Multi-Agent Benchmark

Existing benchmarks assign roles through system prompts that agents can ignore. TeamBench enforces role boundaries at the OS level through Docker container bind mounts.

How it works

Three-Role Pipeline with
Container-Level Isolation

Each role runs in an isolated Docker container with strictly scoped filesystem access. No single container can read the full specification, modify the workspace, and produce the verification attestation.

🔎
Planner
Reads spec.md. Decomposes requirements, identifies edge cases, creates execution plan. Cannot execute code or modify workspace.
↓ sends plan via message
✍️
Executor
Reads brief.md only. Runs commands, modifies workspace code. Cannot access full specification. Follows Planner's instructions.
↓ workspace submitted
✅
Verifier
Reads spec.md + workspace (read-only). Checks compliance, writes attestation.json. Cannot modify workspace or run commands.

Structural Enforcement, Not Prompt Compliance

Unlike prompt-based honor systems where a "Planner" agent retains full technical capability, TeamBench enforces role boundaries through Docker bind mounts. Each container only mounts the directories it needs. Attempting to access a restricted path returns a permission error.

  • ✓ Information Asymmetry by Design — Specification contains details absent from the brief, making Planner relay structurally necessary
  • ✓ Five Ablation Conditions — Systematically remove roles to isolate each component's marginal contribution
  • ✓ Parameterized Generators — 153 generators produce deterministic but distinct instances from different seeds, preventing memorization
  • ✓ Partial Scoring — Deterministic graders produce scores in [0,1], enabling fine-grained comparison
  • ✓ Teamwork Necessity Index — Measures how much of the oracle-vs-restricted gap the team recovers
TNI = (Steam - Srestricted) / max(ε, Soracle - Srestricted)
TNI = 1.0 means full recovery; TNI > 1.0 means team exceeds oracle ceiling

Key Findings

When Does Multi-Agent Collaboration Help?

Team benefit is conditional, not universal. It concentrates on the hardest tasks and weakest models.

⚖️

The Equalizer Effect

Team coordination helps most when single agents fail. The weakest model (Claude Haiku 4.5) shows the largest team uplift, while the strongest models see minimal gains.

+32.1% uplift for weakest model
📈

Planning Drives Value

Planning adds +4.8% on average (p=0.02). The Planner's specification relay guides executors with knowledge they cannot independently access.

+4.8% planning value
⚠️

Verification Hurts

Current LLM verifiers add no net value on average and actively hurt 31% of tasks. The most common failure: false rejection of valid solutions.

-5.8% verification value
🎯

Hardest Tasks Benefit Most

Tasks in the hardest quintile (oracle < 22%) show +15.3% team uplift (p < 0.05). Easy tasks show negative uplift as coordination adds overhead.

+15.3% on Q1 hardest
👥

Universal Across Families

All six closed-source models from three provider families (Google, OpenAI, Anthropic) show positive team uplift. Zero tasks consistently suffer.

6/6 models benefit
🔬

Tool Specialization Works

When roles have specialized tools (Bandit, Mypy, Ruff for Planner; pytest for Verifier), the team achieves 72% vs 32% oracle, with TNI = 1.25.

TNI = 1.25 (exceeds oracle)
