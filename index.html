Three-Role Pipeline with Container Isolation

Each role runs in an isolated Docker container. No single container can access everything.

1

Planner

Reads the full specification (spec.md). Decomposes requirements, identifies hidden constraints and edge cases, creates an execution plan. Cannot execute code or modify the workspace.

2

Generator

Generates the implementation from the Planner's instructions. Runs commands and modifies workspace code. Cannot access the full specification.

3

Evaluator

Evaluates compliance against requirements and writes attestation. Reads spec.md and workspace (read-only). Cannot modify the workspace.

Structural Enforcement, Not Prompt Compliance

Container bind mounts physically enforce role boundaries.

  • Information asymmetry by design — Specification contains critical details absent from the brief
  • Five ablation conditions — Systematically remove roles to isolate marginal contribution
  • Parameterized generators — 153 generators produce distinct instances from different seeds
  • Partial scoring — Deterministic graders produce scores in [0, 1]
TNI = (Steam − Srestricted) / max(ε, Soracle − Srestricted)
TNI = 1.0 means full recovery; TNI > 1.0 means team exceeds oracle ceiling

When Does Multi-Agent Collaboration Help?

Team benefit concentrates on the hardest tasks and weakest models.

The Equalizer Effect

The weakest model (Claude Haiku 4.5) gains the most from team coordination, with +32.1% pass rate uplift — the largest gain across all tested models.

+32.1% weakest model

Planning Drives Value

Planning adds +4.8% on average (p=0.02). The Planner's relay of specification details provides critical knowledge the Generator cannot access through its brief alone.

+4.8% planning

Evaluation Overhead

LLM evaluators add no net value and hurt 31% of tasks through false rejection of valid work. No-Verify outperforms Full Team on 81.5% of tasks — the single strongest finding in the benchmark.

81.5% tasks: No-Verify beats Full Team

Hardest Tasks Benefit Most

Bottom quintile tasks (oracle < 22%) gain +15.3% from the full team. Easy tasks show negative uplift. (driven primarily by the planning component; full three-role team shows smaller gains due to evaluation overhead)

+15.3% on Q1

Generator Bottleneck

Swapping in a stronger Generator boosts score by +0.126 avg — more than any other role upgrade.

+0.126 avg score

Adversarial Tasks Amplify Teamwork

TRAP tasks have the highest team benefit. The Planner catches spec contradictions the Generator never sees.

Get Started

Quick Start

# Install
pip install teambench

# Run on the 28-task Mini subset
teambench run --model your-model --tasks mini --seed 0

# Run full 931-task evaluation
teambench run --model your-model --tasks all --seed 0

# Grade results
teambench grade --run-dir results/

# Use a framework adapter (AutoGen, CrewAI, LangGraph)
teambench run --model your-model --adapter autogen --tasks all --seed 0

To add your model to the leaderboard, submit a Pull Request to the TeamBench repository with your results JSON file in shared/ablation_results/. We recommend evaluating on the TeamBench-Mini TeamBench-Mini subset with all 5 ablation conditions for comparability.

FAQ

Why only one team decomposition (Planner-Generator-Evaluator)?

TeamBench is evaluation infrastructure, not a deployment prescription. The three-role decomposition represents one natural factorization of understanding, implementation, and validation. We additionally evaluate four alternative topologies (Verify-First, Iterative, Dual-Exec, Self-Check) in a topology ablation. Future versions will add multi-round dialogue and router-worker patterns as first-class evaluation modes.

Is the "oracle" baseline fairly optimized?

The oracle is an unrestricted single agent with full access to specification, workspace, and all tools. We tested two enhanced variants (Oracle-CoT and Oracle-2Pass) on the 28-task Mini subset; neither improved over the standard oracle. This establishes that team benefit is structural and cannot be recovered by prompt engineering alone. The oracle label refers to "unrestricted access," not a theoretical upper bound.

Why do open-source models score near zero?

Near-zero performance for the seven open-source models we tested reflects tool-use infrastructure failures (malformed function calls, context overflow at 8K tokens), not model capability limitations. All were served via vLLM with max_model_len=8192. Models with native function-calling support and longer context windows should perform better. We are actively evaluating newer open-source models with improved tool-use capabilities.

How contamination-resistant are the parameterized generators?

Each of the 153 generators randomizes surface parameters (variable names, config values, API field names, bug locations) while preserving structural complexity. Seeds 0-2 are used for public evaluation; seeds 3+ are reserved for hidden holdout. A model that memorizes seed-0 solutions gains no advantage on seed-3 instances because the specific values, names, and locations differ. The 11 static GitHub tasks cannot be parameterized but serve as real-world anchors.

Can labs optimize on the Mini subset to inflate leaderboard scores?

The Mini subset is balanced across all 21 categories and uses parameterized generators. Optimizing for 28 specific task structures is unlikely to transfer to the full 931-task benchmark. For high-stakes comparisons, we recommend the full evaluation or the hidden-seed track (seeds 3+), where the specific task instances have never been publicly evaluated.
